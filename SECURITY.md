# 📖 Supported Versions

The following versions currently receive security updates.

| Version | Supported |
|----------|:---------:|
| Latest Development | ✅ |
| Previous Stable Release | ✅ |
| Older Versions | ❌ |

If you discover a vulnerability in an unsupported version, please verify whether it still exists in the latest version before reporting it.

---

# 🚨 Reporting a Vulnerability

If you discover a security vulnerability, **please do not publish it publicly before it has been reviewed.**

Instead, create a **private security report** or contact the project maintainer with the following information:

- Description of the vulnerability
- Steps to reproduce
- Affected version
- Hardware or emulator used
- Expected behavior
- Actual behavior
- Proof of Concept (if available)
- Possible impact

Providing detailed information helps us investigate and resolve issues more quickly.

---

# 🤝 Responsible Disclosure

We kindly ask security researchers to:

- Report vulnerabilities responsibly.
- Give maintainers reasonable time to investigate and fix the issue.
- Avoid public disclosure before a fix is available.
- Help verify the fix if requested.

Responsible disclosure helps protect all users of NexOS.

---

# 🔄 Security Response Process

Every security report will follow these steps:

1. Report received.
2. Initial review.
3. Vulnerability verification.
4. Fix development.
5. Security testing.
6. Patch release.
7. Public disclosure (if appropriate).

---

# 🛡 Security Best Practices

When contributing to NexOS:

- Validate all external input.
- Avoid undefined behavior.
- Check memory boundaries.
- Handle errors correctly.
- Write secure and readable code.
- Review security-sensitive changes carefully.
- Test before submitting Pull Requests.

---

# 🎯 Scope

Security reports may include issues related to:

- Kernel Memory Management
- Buffer Overflows
- Privilege Escalation
- Interrupt Handling
- Device Drivers
- Networking Stack
- File System
- Bootloader
- System Calls
- Build System
